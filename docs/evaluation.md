# Development Evaluation

## Goal

Evaluate the Vietnamese receipt OCR and information extraction pipeline on the current MVP development benchmark.

The evaluation compares manually annotated ground-truth JSON files against extracted JSON files generated by the OCR + parser pipeline.

## Dataset

Current development benchmark:

- 15 Vietnamese receipt/invoice images
- 15 manual ground-truth JSON files
- 39 ground-truth item rows
- Multiple receipt formats, including retail, bookstore, coffee shop, restaurant, and small shop receipts

These receipts were used during parser development and error analysis. The results below are therefore development-set results, not held-out test results. The dataset is intentionally small and is not a production-scale benchmark.

The full 15-receipt dataset, predictions, and generated evaluation summaries are private and ignored by Git. A fresh public checkout contains one anonymized sample for a smoke test, but it cannot reproduce the aggregate metrics below. The reported metrics are documented snapshots of the local development experiment.

## Metric Semantics

- Receipt-field accuracy is the proportion of correct field decisions across six fields. Store names pass at normalized similarity `>= 0.75`; invoice IDs pass on exact normalized match or similarity `>= 0.85`; datetime can pass when the date matches even if time does not.
- Item rows are aligned by receipt order, not by an assignment or detection-matching algorithm.
- Item names pass at normalized similarity `>= 0.75`.
- Quantity, unit price, and line total use exact equality after number normalization.
- Overall item-field accuracy is the unweighted mean of name, quantity, unit-price, and line-total accuracies. Item-count accuracy is reported separately.

## Reproduce Evaluation

The commands below reproduce the evaluation only when the private local development images, labels, and predictions are present. From a public checkout, use the tracked sample to smoke-test the pipeline instead; the published aggregate metrics are not independently reproducible yet.

Run the full receipt-level pipeline:

```powershell
python scripts/run_ocr.py --all
python scripts/run_extraction.py --all
python scripts/evaluate_extraction.py --all --split development
```

Evaluate default text-based item extraction:

```powershell
python scripts/evaluate_items.py --split development
```

Run and evaluate layout-aware item extraction:

```powershell
python scripts/batch_inspect_ocr_layout.py
python scripts/run_layout_item_extraction.py --all
python scripts/evaluate_layout_items.py --split development
```

Evaluation outputs:

```text
data/evaluation/development/evaluation_report.csv
data/evaluation/development/evaluation_summary.json
data/evaluation/development/item_evaluation_report.csv
data/evaluation/development/item_evaluation_summary.json
data/evaluation/development/item_evaluation.md
data/evaluation/development/layout_aware_item_evaluation_report.csv
data/evaluation/development/layout_aware_item_evaluation_summary.json
data/evaluation/development/layout_aware_item_evaluation.md
```

All three evaluator entry points default to `development`, select receipt IDs from `data/dataset_manifest.csv`, and write to a split-specific directory. Use `--split held_out` only for frozen receipts that were not used to develop parser rules. If a requested split has no rows, evaluation stops without writing an empty report.

A receipt-level run with `--receipt-id` writes under `data/evaluation/<split>/single/<receipt_id>/`, so it cannot overwrite the full-split report.

## Observations

### Strong fields

`payment_method` and `items_count` are currently correct on the MVP development benchmark. `total_amount` and `datetime` are also strong after improving Vietnamese money normalization and date/time parsing.

### Fields that still need improvement

`store_name` and `invoice_id` remain the most error-prone receipt-level fields.

Common causes include:

- OCR distortion in receipt headers
- Short or stylized store names
- invoice IDs mixed with cashier IDs, phone numbers, order numbers, or transaction IDs
- receipt codes split across multiple OCR lines

### Item extraction

The text-based parser is limited when item names, quantities, unit prices, and line totals are split across multiple OCR lines.

The layout-aware parser improves item extraction on the MVP development benchmark by using OCR bounding-box row structure instead of relying only on raw OCR text order.

## Known Limitations

- OCR errors propagate directly into the parser.
- The default parser is rule-based and sensitive to layout variation.
- The layout-aware parser has only been evaluated on the same data used for development and error analysis.
- The current dataset is small and not representative of all Vietnamese receipt formats.
- The reported 100.00% layout-aware item result is not a held-out estimate and should not be treated as production-level generalization.
- The private benchmark and generated artifacts are not committed, so the aggregate results cannot be reproduced from a fresh public checkout.

## Next Improvements

Recommended next steps:

1. Validate the layout-aware item parser on more receipt formats.
2. Add more sample receipts with different layouts and image qualities.
3. Improve `store_name` and `invoice_id` extraction.
4. Add confidence-based warnings for suspicious predictions.
5. Add a small reproducible sample dataset for GitHub users.
6. Add basic unit tests for parser and normalizer utilities.
7. Consider a lightweight FastAPI endpoint after the repo is stable.
