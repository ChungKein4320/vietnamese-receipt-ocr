# Evaluation

## Goal

Evaluate the rule-based information extraction pipeline on a small Vietnamese receipt dataset.

The evaluation compares manually annotated ground truth JSON files against extracted JSON files generated by the OCR and parser pipeline.

## Dataset

Current evaluation dataset:

* 15 Vietnamese receipt/invoice images
* 15 manual ground truth JSON files
* Multiple receipt formats:

  * retail receipts
  * bookstore receipts
  * coffee shop receipts
  * restaurant receipts
  * small shop receipts

## Pipeline Version

Current parser version:

```text
rule_based_v0.1
```

Pipeline:

```text
image
→ PaddleOCR
→ raw OCR text
→ text normalization
→ rule-based field extraction
→ structured JSON
→ field-level evaluation
```

## Evaluated Fields

The current evaluator measures:

| Field            | Description               |
| ---------------- | ------------------------- |
| `store_name`     | Store or business name    |
| `datetime`       | Receipt date/time         |
| `invoice_id`     | Invoice or receipt code   |
| `total_amount`   | Final payable amount      |
| `payment_method` | Payment method            |
| `items_count`    | Number of extracted items |

The evaluator currently checks item count only. It does not yet evaluate item-level name, quantity, unit price, or line total accuracy.

## Observations

### Strong fields

`total_amount` performs best after improving money normalization. The parser can now handle values such as:

```text
70.000d
80.000d
20.000d
CASH(VND)-88000
```

### Medium fields

`store_name` and `datetime` are reasonably stable, but OCR errors and missing time values still affect accuracy.

### Weak fields

`invoice_id` is the weakest field because receipt IDs have many inconsistent formats and are often confused with phone numbers, invoice titles, cashier IDs, or transaction codes.

### Item extraction

`items_count` remains challenging because receipt layouts differ significantly. Some receipts split item names, quantities, unit prices, and line totals across separate OCR lines.

## Known Limitations

* OCR errors propagate directly into the parser.
* Rule-based extraction is sensitive to layout variation.
* Item-level evaluation is not implemented yet.
* Invoice ID extraction requires more robust pattern handling.
* Payment method can be missed when OCR reads `Tiền mặt` incorrectly.
* The current dataset is small and intended for MVP evaluation, not production benchmarking.

## Error Analysis

### Total amount

`total_amount` is the strongest field after improving the money regex to support Vietnamese-style currency suffixes such as `d`, `đ`, and `VND`.

Remaining errors are likely caused by:

* OCR reading the final amount incorrectly
* discount lines being confused with payable amount
* receipt layouts where total amount is separated from its keyword

### Store name

Store name extraction is mostly correct, but can fail when:

* OCR reads the business name poorly
* address lines appear before the actual store name
* the receipt header is heavily distorted
* the business name is short or stylized

### Datetime

Datetime extraction works when the date follows common formats such as:

```text
15/08/2020 10:28
04.10.2020 16.21
2020-08-15 09:47
```

Failures happen when:

* OCR corrupts separators
* time appears before date
* date and time are split across multiple lines
* the date is partially unreadable

### Invoice ID

Invoice ID extraction is currently weak because receipt codes have highly inconsistent formats.

Examples of difficult cases:

* receipt IDs that look like phone numbers
* receipt IDs mixed with cashier codes
* receipt IDs with prefixes removed by OCR
* transaction IDs split across multiple OCR lines

### Payment method

Payment method extraction currently uses keyword matching.

Supported normalized values:

```text
cash
card
bank_transfer
e_wallet
unknown
```

Failures happen when OCR misreads payment keywords such as `Tiền mặt`.

### Items count

Item extraction is the hardest part of the current rule-based parser.

Common failure cases:

* item rows split across multiple OCR lines
* barcode lines between item name and price
* discount rows detected as items
* subtotal rows detected as items
* add-ons or toppings counted differently from ground truth

## Next Improvements

Potential improvements:

1. Add item-level evaluation.
2. Improve invoice ID extraction.
3. Add layout-aware item parsing using OCR bounding boxes.
4. Add confidence-based warnings for suspicious predictions.
5. Compare PaddleOCR with VietOCR.
6. Add optional LLM-based parser for difficult receipts.
7. Expand the evaluation dataset beyond 15 receipts.
